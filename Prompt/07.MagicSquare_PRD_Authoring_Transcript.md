# Magic Square 4x4 — PRD 작성·검토 Prompting Transcript Export

아래는 PRD 작성 세션의 사용자-어시스턴트 대화를 순서대로 정리한 export transcript이다.

---

## 세션 범위

- Turn 1: PRD 참고 문서 분석 및 섹션 매핑
- Turn 2: PRD 본문 작성 (23개 섹션)
- Turn 3: PRD 품질 검토 (7개 기준)
- Turn 4: PRD 파일 저장 (`docs/PRD_MagicSquare.md`)
- Turn 5: Report·Prompting Transcript Export

---

## Turn 1 — PRD 참고 문서 분석 및 매핑

### User

PRD 작성 전 참고 문서 8종을 분석하고, 각 문서를 PRD 섹션에 매핑해 달라고 요청함.

- 구현 코드·테스트 코드·파일 수정·PRD 본문 작성 **금지**
- 출력 형식: Overall Judgment, Document Summary, Section Mapping, Source Priority, Direct/Appendix 구분, Conflicts, Recommended Outline

참고 문서:

1. `Report/01.4x4_MagicSquare_Problem_Definition_Report.md`
2. `Report/02.4x4_MagicSquare_TDD_Design_Report.md`
3. `Report/03.4x4_MagicSquare_CursorRules_UserEntity_Implementation_Report.md`
4. `Report/04.4x4_MagicSquare_CursorRules_MDC_Migration_Report.md`
5. `Report/05.Agent_Definitions_Standardization_Report.md`
6. `Report/06.MagicSquare_Level1-5_Scenario_Verification_Report.md`
7. `.cursorrules`
8. `.cursor/rules/*.mdc`

### Assistant

8개 문서를 읽고 **PRD Reference Document Analysis**를 출력함.

주요 판단:

- **PRD 작성 가능 여부:** 조건부 가능 (약 75%)
- **1차 참고:** Report/02, Report/06, Report/01
- **Report/04 ≠ Vision/Epic 문서** — 실제로 MDC Migration 보고서; Epic/User Journey는 Report/06 + Level 1~5 산출물
- **누락:** Level 1~4 전문, Persona/Journey/Story/Gherkin 본문 파일
- 권장 PRD 목차 21개 항목 제시

---

## Turn 2 — PRD 본문 작성

### User

Senior Product/Software Architect 역할로 Magic Square 4x4 **구현 전 PRD** 작성을 요청함.

- 구현·테스트 코드·범위 확장·모호 표현 금지
- Dual-Track (Track A: Boundary, Track B: Domain) 반드시 반영
- Concept → Rule → Use Case → Contract → Test → Component Traceability 포함
- 파일 생성 금지, Markdown 본문만 출력
- 23개 섹션 구조 준수 (Executive Summary ~ Appendix)
- 고정 I/O 계약, Solver Attempt 1/2, both-fail 정책 확정 요구

참고 문서 우선순위:

- Report/4 → Vision, Epic, User Story, AC, Gherkin
- Report/1 → Background, Problem
- Report/2 → Functional Scope, Contract, Invariant
- Report/3 + Cursor Rules → Quality, Engineering

### Assistant

**PRD — Magic Square 4x4 TDD Practice** 전문(23개 섹션)을 Markdown으로 출력함.

핵심 산출:

- FR-01~FR-05, BR-01~BR-15
- §12 Input/Output Contract, §13 Error Policy
- §15 Dual-Track TDD Strategy
- §16 Test Plan (Matrix A small-first, Matrix B reverse)
- §21 Traceability Matrix
- §22 Open Questions (DN-01~DN-05)
- both-fail → `DOMAIN-ERR-NO_VALID_PLACEMENT` 구조화 실패 응답 **확정**

## Turn 3 — PRD 품질 검토

### User

작성된 PRD를 7개 기준으로 검토해 달라고 요청함.

1. FR별 테스트 가능 AC
2. AC ↔ Traceability Matrix 연결
3. Boundary/Domain 책임 혼합
4. 오류 정책 미정 여부
5. small-first / reverse 테스트 데이터 구분
6. 1-index / row-major 누락
7. 구현·테스트 코드 포함 여부

- **수정하지 말고** 문제와 개선안만 보고

### Assistant

**PRD 검토 보고서** 출력 (수정 없음).

종합:

| 기준 | 판정 |
|---|---|
| 1 FR별 AC | ⚠️ FR-05-06 등 보강 필요 |
| 2 Traceability | ❌ 다수 AC Matrix 미연결 |
| 3 Layer 분리 | ⚠️ FR-05 통합, resolver 용어 모호 |
| 4 오류 정책 | ⚠️ DN-01, UI-ERR-005/006 공백 |
| 5 테스트 데이터 | ✅ Matrix A/B 구분 |
| 6 row-major/1-index | ⚠️ BR-16(출력 blank 순서) 부재 |
| 7 코드 미포함 | ✅ |

우선 개선 권고: DN-01 결정 → AC 전수 Traceability → BR-16 → FR-05 Split → both-fail fixture
