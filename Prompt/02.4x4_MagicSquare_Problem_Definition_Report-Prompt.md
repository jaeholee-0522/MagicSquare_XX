# Magic Square 4x4 Prompt Transcript Export

- 목적: TDD 설계 문서 생성을 위한 실행 프롬프트 정제

## Turn 1 - User
너는 Cursor.AI용 프롬프트 설계자다.  
나는 Magic Square 4x4 TDD 설계 문서를 만들기 위한 프롬프트가 필요하다.  
지금 설계 문서를 만들지 말고, 그 설계 문서를 만들 수 있는 실행용 프롬프트만 작성하라.

## Turn 1 - Assistant (초안 프롬프트)
```text
[역할]
너는 시니어 소프트웨어 아키텍트이자 TDD 코치다.
목표는 “4x4 Magic Square” 문제를 구현하기 위한 TDD 설계 문서를 작성하는 것이다.

[중요 지시]
- 지금 단계에서 코드를 작성하지 말고, 설계 문서만 작성하라.
- 설명은 한국어로 작성하라.
- 문서는 실무에서 바로 구현 가능한 수준으로 구체적으로 작성하라.
- 근거 없는 일반론은 피하고, 4x4 Magic Square 문제에 직접 연결된 내용만 작성하라.

[문서 목적]
- 개발자가 문서만 보고도 TDD 방식으로 구현을 시작할 수 있어야 한다.
- 테스트 우선 개발(RED-GREEN-REFACTOR) 흐름이 명확히 드러나야 한다.

[필수 포함 항목]
1) 문제 정의
- 4x4 매직스퀘어의 수학적 조건(행/열/대각선 합, 사용 숫자 범위, 중복 금지 등)
- 입력/출력/성공 기준(acceptance criteria)
- 비기능 요구사항(성능, 확장성, 가독성 등 최소 항목)

2) 도메인 모델 및 용어
- 핵심 개념(보드, 셀, 후보군, 제약조건 등) 정의
- 불변조건(invariants) 명시

3) 아키텍처/모듈 설계
- 책임 분리(예: 생성기, 검증기, 상태관리, 유틸)
- 각 모듈의 역할, 인터페이스 초안(함수/클래스 시그니처 수준)
- 의존 관계와 확장 포인트

4) TDD 전략
- RED-GREEN-REFACTOR 사이클을 단계별로 제시
- 작은 단위 테스트부터 통합/시나리오 테스트로 확장하는 순서 제안
- 테스트 우선순위(핵심 규칙 → 제약 검증 → 생성 알고리즘 → 예외/경계)

5) 테스트 설계 상세
- 테스트 분류(Unit/Integration/Property/Regression)
- Given-When-Then 형식의 구체적 테스트 케이스 목록
- 정상/오류/경계/퇴행 케이스 포함
- 실패 메시지 전략(디버깅 가능한 assertion 관점)

6) 알고리즘 설계 대안 비교
- 최소 2개 접근(예: 백트래킹 기반, 제약 전파/휴리스틱 기반)
- 시간복잡도/구현 난이도/테스트 용이성 비교
- 최종 채택안과 채택 근거

7) 리스크 및 완화
- 구현 시 예상 리스크(탐색 폭증, 잘못된 pruning, 테스트 취약점 등)
- 리스크별 탐지/완화 전략

8) 구현 로드맵
- 단계별 작업 계획(Phase 단위)
- 각 단계의 완료 조건(Definition of Done)
- 문서 기반으로 바로 작업 티켓을 만들 수 있게 체크리스트 제공

[출력 형식]
- Markdown으로 작성
- H2/H3 구조를 사용해 섹션 구분
- 표가 필요하면 간결하게 사용
- 마지막에 “즉시 실행 가능한 첫 5개 테스트”를 번호 목록으로 제시

[품질 기준]
- 모호한 표현(“적절히”, “필요시”)을 지양하고, 측정 가능한 기준으로 작성
- 중복 설명 제거, 섹션 간 추적 가능성 유지(요구사항 ↔ 테스트 ↔ 모듈)
- 실제 구현자가 바로 착수 가능한 실천 중심 문서여야 함

이제 위 기준을 충족하는 “Magic Square 4x4 TDD 설계 문서”를 작성하라.
```

## Turn 2 - Assistant (개정 프롬프트)
```text
[역할]
당신은 Dual-Track UI + Logic TDD 및 Clean Architecture 설계 전문가입니다.

[프로젝트]
- 이름: Magic Square (4x4)
- 성격: TDD 연습용
- 핵심 목적: 알고리즘 난이도 자체보다 “레이어 분리 + 계약 기반 테스트 + 리팩토링” 훈련

[절대 제약]
- 구현 코드는 작성하지 마십시오. (설계/계약/테스트/통합 계획만 작성)
- UI는 실제 화면이 아니라 입력/출력 경계(Boundary)로 정의하십시오.
- Data Layer는 DB 구현이 아니라 저장/로드 인터페이스 수준으로 정의하십시오. (메모리/파일 교체 가능)
- 입력/출력 계약은 아래 명세로 고정하십시오.
- 모호한 표현 금지: “적절히”, “충분히”, “필요시” 같은 표현 사용 금지
- 모든 규칙은 테스트로 검증 가능해야 합니다.
- 표/체크리스트를 적극 사용하십시오.

[입력 계약(고정)]
- 타입: 4x4 int[][]
- 0은 빈칸
- 빈칸은 정확히 2개
- 값 범위: 0 또는 1~16
- 0 제외 중복 금지

[출력 계약(고정)]
- 타입: int[6]
- 좌표는 1-index
- 반환 형식: [r1,c1,n1,r2,c2,n2]
- n1,n2는 두 누락 숫자이며:
  - (작은수→첫 빈칸, 큰수→둘째 빈칸) 조합이 마방진이면 그 순서를 채택
  - 아니면 반대 순서를 채택

[문서 작성 목적]
- 문서만 보고 바로 TDD 구현을 시작할 수 있어야 합니다.
- RED-GREEN-REFACTOR 흐름이 단계적으로 명확해야 합니다.
- 요구사항 ↔ 계약 ↔ 테스트 ↔ 컴포넌트가 추적 가능해야 합니다.

------------------------------------------------------------
출력 형식 (반드시 아래 구조/제목 그대로 사용)
------------------------------------------------------------
# 1) Logic Layer (Domain Layer) 설계
## 1.1 도메인 개념
- Entities / Value Objects / Domain Services 목록과 책임(SRP)

## 1.2 도메인 불변조건(Invariants)
- 행/열/대각선 합 일치, Magic Constant 등

## 1.3 핵심 유스케이스(도메인 관점)
- 빈칸 찾기, 누락 숫자 찾기, 마방진 판정, 두 조합 시도

## 1.4 Domain API(내부 계약)
- 메서드 시그니처 수준(코드 X) + 입력/출력/실패조건

## 1.5 Domain 단위 테스트 설계(RED 우선)
- 테스트 케이스 목록(정상/비정상/엣지)
- 각 테스트가 보호하는 invariant 명시

# 2) Screen Layer (UI Layer) 설계 (Boundary Layer)
## 2.1 사용자/호출자 관점 시나리오
- “행렬 입력 → 검증 → 결과 출력” 흐름

## 2.2 UI 계약(외부 계약)
- Input schema / Output schema / Error schema

## 2.3 UI 레벨 테스트(Contract-first, RED 우선)
- 잘못된 크기, 빈칸 개수 오류, 값 범위 오류, 중복 오류, 반환 포맷 검증
- Domain은 Mock으로 가정

## 2.4 UX/출력 규칙
- 에러 메시지 표준(정확한 문구 규칙까지)

# 3) Data Layer 설계 (Data Layer)
## 3.1 목적 정의
- “저장/로드”의 필요성과 범위(학습용)

## 3.2 인터페이스 계약
- 예: MatrixRepository.save/load (메서드 수준, 코드 X)
- 저장 대상: 입력 행렬, 실행 결과(선택)

## 3.3 구현 옵션 비교(메모리/파일)
- 옵션 A: InMemory / 옵션 B: File(JSON/CSV)
- 추천안 1개 선택 + 이유

## 3.4 Data 레이어 테스트
- 저장/로드 정합성, 예외(파일 없음/형식 오류), 불변조건(4x4 유지)

# 4) Integration & Verification (통합 및 검증)
## 4.1 통합 경로 정의
- UI → Application(선택) → Domain → Data 흐름(의존성 방향 포함)

## 4.2 통합 테스트 시나리오
- 정상 시나리오 2개 이상
- 실패 시나리오 3개 이상(입력 오류, 도메인 실패, 데이터 실패)

## 4.3 회귀 보호 규칙
- 기존 테스트 유지 정책
- 변경 금지 규칙(계약/출력 포맷)

## 4.4 커버리지 목표
- Domain Logic 95%+
- UI Boundary 85%+
- Data 80%+

## 4.5 Traceability Matrix (필수)
- Concept(Invariant) → Rule → Use Case → Contract → Test → Component

------------------------------------------------------------
작성 규칙
------------------------------------------------------------
- Markdown으로 작성하십시오.
- 각 섹션에서 표와 체크리스트를 포함하십시오.
- 모든 규칙/정책은 “검증 방법(어떤 테스트로 확인하는지)”을 함께 적으십시오.
- 구현 코드, 의사코드, 실제 언어 문법 코드는 금지합니다.
- 마지막에는 “즉시 RED로 시작할 첫 10개 테스트”를 번호 목록으로 작성하십시오.
- 문서는 한국어로 작성하십시오.

이제 위 조건을 100% 준수하는 “Magic Square 4x4 TDD 설계 문서”를 작성하십시오.
```
