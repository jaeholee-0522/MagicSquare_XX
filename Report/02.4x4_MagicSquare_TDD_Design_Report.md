# Magic Square 4x4 TDD 설계 문서

## 1) Logic Layer (Domain Layer) 설계

### 1.1 도메인 개념

| 분류 | 이름 | 책임(SRP) | 입력 | 출력 | 검증 방법 |
|---|---|---|---|---|---|
| Entity | Matrix4x4 | 4x4 숫자 상태를 보존하고 좌표 접근을 제공 | 4x4 값 집합 | 좌표별 값 조회 결과 | 행/열 크기 검증 단위 테스트 |
| Value Object | CellPosition | 1-index 좌표 규칙을 보장 | row, col | 유효 좌표 객체 | 좌표 범위(1~4) 실패 테스트 |
| Value Object | MissingCells | 정확히 2개 빈칸 좌표를 표현 | 0의 좌표 목록 | 정렬된 2개 좌표 | 빈칸 1개/3개 입력 실패 테스트 |
| Value Object | MissingNumbers | 누락 숫자 2개를 표현(오름차순) | 1~16 사용 현황 | 정렬된 2개 숫자 | 중복/범위 오류 테스트 |
| Domain Service | MagicSquareValidator | 마방진 불변조건 검증 | Matrix4x4 | 판정 결과(성공/실패 사유) | 합/중복/범위 위반 테스트 |
| Domain Service | MissingPairResolver | 두 누락 숫자의 배치 순서를 결정 | MissingCells, MissingNumbers, Matrix4x4 | [r1,c1,n1,r2,c2,n2] | 두 조합 시도 결과 비교 테스트 |

체크리스트:
- [ ] Entity는 상태 보존 책임만 가진다.
- [ ] Value Object는 생성 시점에 자체 유효성 검증을 수행한다.
- [ ] Domain Service는 상태 저장을 하지 않는다.
- [ ] 결과 포맷 책임은 Domain Service에서 완결된다.

### 1.2 도메인 불변조건(Invariants)

| ID | 불변조건 | 규칙 정의 | 실패 조건 | 검증 방법 |
|---|---|---|---|---|
| INV-01 | 크기 불변 | 행=4, 열=4 | 4x4가 아님 | 크기 오류 단위 테스트 |
| INV-02 | 값 범위 불변 | 모든 값은 0 또는 1~16 | 음수, 17 이상 | 범위 오류 단위 테스트 |
| INV-03 | 빈칸 개수 불변 | 값 0은 정확히 2개 | 0의 개수 ≠ 2 | 빈칸 개수 단위 테스트 |
| INV-04 | 중복 불변 | 0 제외 값 중복 금지 | 같은 숫자 2회 이상 | 중복 검출 단위 테스트 |
| INV-05 | 매직 상수 | 4x4 매직 상수는 34 | 행/열/대각선 합이 34 아님 | 합 검증 단위 테스트 |
| INV-06 | 출력 좌표 불변 | 결과 좌표는 1-index, 1~4 | 0-index/범위 초과 | 출력 계약 테스트 |
| INV-07 | 배치 순서 불변 | 작은수-첫빈칸 우선 시도, 실패 시 반전 | 순서 결정 규칙 미준수 | 순서 결정 분기 테스트 |

체크리스트:
- [ ] 모든 Invariant는 독립 테스트 가능하다.
- [ ] 실패 사유는 단일 원인으로 분류된다.
- [ ] 도메인 검증 순서는 재현 가능하도록 고정된다.

### 1.3 핵심 유스케이스(도메인 관점)

| Use Case ID | 유스케이스 | 사전조건 | 처리 규칙 | 산출물 | 검증 방법 |
|---|---|---|---|---|---|
| UC-D-01 | 빈칸 찾기 | INV-01~04 충족 입력 | 0의 좌표 2개 추출 | MissingCells | 좌표 개수/순서 테스트 |
| UC-D-02 | 누락 숫자 찾기 | INV-01~04 충족 입력 | 1~16 중 미사용 2개 도출 | MissingNumbers | 누락 숫자 정확성 테스트 |
| UC-D-03 | 마방진 판정 | 후보 배치 완료 상태 | 행/열/대각선 합=34 검사 | 참/거짓 + 사유 | 합 불일치 테스트 |
| UC-D-04 | 두 조합 시도 | MissingCells, MissingNumbers 확보 | 조합 A 후 실패 시 조합 B | 최종 int[6] | 조합 전환 테스트 |

체크리스트:
- [ ] 각 유스케이스는 단일 책임을 유지한다.
- [ ] 유스케이스 간 데이터 전달 객체가 명확하다.
- [ ] 실패는 예외가 아닌 도메인 실패 결과로 표준화한다.

### 1.4 Domain API(내부 계약)

| API 이름 | 입력 계약 | 출력 계약 | 실패조건 | 검증 방법 |
|---|---|---|---|---|
| ValidateMatrix | 4x4 int[][] | ValidatedMatrix 또는 ValidationError | INV-01~04 위반 | 입력 검증 테스트 |
| LocateMissingCells | ValidatedMatrix | MissingCells(2개 좌표) | 빈칸 2개 아님 | 빈칸 개수 테스트 |
| DeriveMissingNumbers | ValidatedMatrix | MissingNumbers(2개 숫자) | 누락 숫자 계산 불가 | 누락 숫자 테스트 |
| EvaluateMagicSquare | CandidateMatrix | MagicCheckResult | 합 불일치 | 행/열/대각선 테스트 |
| ResolvePlacementOrder | ValidatedMatrix + MissingCells + MissingNumbers | int[6] | 두 조합 모두 실패 | 조합 A/B 분기 테스트 |

체크리스트:
- [ ] API 단위 입출력 타입은 고정 계약으로 관리한다.
- [ ] 실패조건은 에러 코드로 식별 가능해야 한다.
- [ ] 출력 int[6] 포맷은 내부 API 최종 단계에서만 생성한다.

## 2) Screen Layer (UI Layer) 설계 (Boundary Layer)

### 2.1 사용자/호출자 관점 시나리오

| 단계 | Boundary 동작 | 입력 | 출력 | 검증 방법 |
|---|---|---|---|---|
| S1 | 입력 수신 | 4x4 int[][] | 입력 수신 확인 | 요청 수신 계약 테스트 |
| S2 | 입력 검증 위임 | 입력 행렬 | 검증 성공/오류 | 오류 코드 매핑 테스트 |
| S3 | 도메인 실행 위임 | 검증 통과 입력 | int[6] 또는 도메인 오류 | Domain Mock 계약 테스트 |
| S4 | 응답 포맷 반환 | 도메인 결과 | Output schema | 포맷 일치 테스트 |

체크리스트:
- [ ] Screen Layer는 계산 로직을 포함하지 않는다.
- [ ] 검증 실패 시 Domain 호출을 차단한다.
- [ ] 성공/실패 응답 스키마가 상호 배타적이다.

### 2.2 UI 계약(외부 계약)

| 계약 구분 | 필드 | 규칙 | 예외 코드 | 검증 방법 |
|---|---|---|---|---|
| Input schema | matrix | 4x4 int[][] 고정 | UI-ERR-001 | 크기 계약 테스트 |
| Input schema | values | 0 또는 1~16 | UI-ERR-002 | 범위 계약 테스트 |
| Input schema | zeros | 정확히 2개 | UI-ERR-003 | 빈칸 개수 테스트 |
| Input schema | duplicates | 0 제외 중복 금지 | UI-ERR-004 | 중복 계약 테스트 |
| Output schema | result | int[6] 고정 길이 | UI-ERR-005 | 길이/타입 테스트 |
| Output schema | coordinate | 1-index (1~4) | UI-ERR-006 | 좌표 범위 테스트 |
| Error schema | message | 표준 문구 정책 준수 | UI-ERR-000 | 메시지 포맷 테스트 |

체크리스트:
- [ ] Input/Output/Error schema를 버전 태그로 관리한다.
- [ ] 에러 코드는 중복 없이 1:1 규칙 대응한다.
- [ ] 성공 응답에는 에러 필드가 없어야 한다.

### 2.3 UI 레벨 테스트(Contract-first, RED 우선)

| 테스트 ID | Domain Mock 동작 | 입력 | 기대 응답 | 검증 방법 |
|---|---|---|---|---|
| UT-01 | 호출 안 됨 | 3x4 입력 | UI-ERR-001 | 검증 선차단 테스트 |
| UT-02 | 호출 안 됨 | 0 개수 1개 | UI-ERR-003 | 빈칸 규칙 테스트 |
| UT-03 | 호출 안 됨 | 값 17 포함 | UI-ERR-002 | 범위 규칙 테스트 |
| UT-04 | 호출 안 됨 | 중복 값 포함 | UI-ERR-004 | 중복 규칙 테스트 |
| UT-05 | 성공 결과 반환 | 정상 입력 | 길이 6 결과 | Output schema 테스트 |
| UT-06 | 도메인 실패 반환 | 정상 형식/불성립 조건 | 표준 도메인 오류 매핑 | 오류 매핑 테스트 |

체크리스트:
- [ ] Contract-first: 테스트를 계약보다 먼저 수정하지 않는다.
- [ ] Mock 검증으로 Boundary 책임을 고립한다.
- [ ] UI 테스트는 계산 정답이 아니라 계약 준수만 검증한다.

### 2.4 UX/출력 규칙

| 항목 | 규칙 | 금지 사항 | 검증 방법 |
|---|---|---|---|
| 에러 메시지 형식 | [에러코드] 원인:조치 | 자유 서술형 문장 | 메시지 정규식 테스트 |
| 성공 메시지 형식 | RESULT_OK + int[6] | 추가 설명 텍스트 | 성공 응답 스키마 테스트 |
| 다중 오류 처리 | 첫 번째 위반 규칙 1건만 반환 | 여러 오류 병합 반환 | 우선순위 테스트 |
| 로깅 분리 | 사용자 메시지와 내부 로그 분리 | 내부 예외 원문 노출 | 민감정보 노출 테스트 |

체크리스트:
- [ ] 사용자 메시지 문구는 고정 템플릿으로 관리한다.
- [ ] 동일 오류 입력은 항상 동일 코드/문구를 반환한다.
- [ ] 경계 계층은 내부 스택 정보 노출을 금지한다.

## 3) Data Layer 설계 (Data Layer)

### 3.1 목적 정의

| 항목 | 정의 | 범위 | 비범위 | 검증 방법 |
|---|---|---|---|---|
| 저장 목적 | TDD 회귀 재현용 입력/출력 기록 | 입력 행렬, 결과 배열 | DB 트랜잭션 | 저장-재실행 일치 테스트 |
| 로드 목적 | 실패 케이스 재현 | 단건/목록 로드 | 실시간 동시성 제어 | 로드 재현 테스트 |
| 학습 목적 | 리포지토리 추상화 훈련 | 인터페이스 중심 | 성능 튜닝 | 계약 테스트 |

체크리스트:
- [ ] Data Layer는 도메인 규칙을 재판정하지 않는다.
- [ ] 저장 단위는 계약 객체 단위로 고정한다.
- [ ] 학습 범위를 넘는 기능(권한, 분산락)은 제외한다.

### 3.2 인터페이스 계약

| 인터페이스 | 입력 | 출력 | 실패조건 | 검증 방법 |
|---|---|---|---|---|
| MatrixRepository.SaveInput | 입력 행렬(4x4) + 요청 식별자 | 저장 성공 식별자 | 스키마 위반/저장 실패 | 저장 계약 테스트 |
| MatrixRepository.LoadInput | 요청 식별자 | 입력 행렬(4x4) | 식별자 없음 | 미존재 식별자 테스트 |
| MatrixRepository.SaveResult | 요청 식별자 + int[6] | 저장 성공 여부 | 포맷 위반/저장 실패 | 결과 포맷 저장 테스트 |
| MatrixRepository.LoadResult | 요청 식별자 | int[6] | 결과 없음/형식 손상 | 손상 데이터 테스트 |

체크리스트:
- [ ] 인터페이스는 구현체 교체 가능해야 한다.
- [ ] 저장 전후 데이터 형태가 동일해야 한다.
- [ ] 실패는 데이터 계층 에러 코드로 구분한다.

### 3.3 구현 옵션 비교(메모리/파일)

| 옵션 | 장점 | 단점 | 테스트 용이성 | 추천 여부 |
|---|---|---|---|---|
| A. InMemory | 빠른 실행, 설정 불필요 | 프로세스 종료 시 데이터 소실 | 매우 높음 | 보조 |
| B. File(JSON/CSV) | 재현성 높음, 기록 보존 | 파일 I/O 오류 처리 필요 | 높음 | 주 추천 |

추천안: 옵션 B(File)

선정 이유:
1. TDD 회귀 재현에서 실패 입력/출력 이력 보존이 가능하다.
2. InMemory 대비 세션 종료 후에도 동일 테스트 자산을 재사용할 수 있다.
3. 인터페이스 분리 상태에서 InMemory를 테스트 더블로 병행하기 쉽다.

검증 방법:
- 파일 저장 후 재로드 일치 테스트
- 파일 없음/손상/권한 오류 테스트
- 동일 식별자 덮어쓰기 정책 테스트

체크리스트:
- [ ] 직렬화 포맷 버전을 명시한다.
- [ ] 파일 경로 정책(상대/절대, 권한)을 고정한다.
- [ ] 테스트에서 임시 저장소를 사용해 격리한다.

### 3.4 Data 레이어 테스트

| 테스트 ID | 시나리오 | 기대 결과 | 검증 방법 |
|---|---|---|---|
| DTL-01 | 입력 저장 후 로드 | 원본 4x4 완전 동일 | Deep equality 테스트 |
| DTL-02 | 결과 저장 후 로드 | 원본 int[6] 완전 동일 | 길이/순서 테스트 |
| DTL-03 | 파일 없음 상태 로드 | DATA-ERR-NOT_FOUND | 예외 매핑 테스트 |
| DTL-04 | 손상 파일 로드 | DATA-ERR-FORMAT_INVALID | 파싱 실패 테스트 |
| DTL-05 | 4x4 아닌 데이터 저장 시도 | 저장 거부 | 불변조건 보존 테스트 |

체크리스트:
- [ ] I/O 실패 테스트는 성공 케이스와 분리한다.
- [ ] 정합성 테스트는 입력과 결과를 독립 검증한다.
- [ ] 데이터 오류 코드는 UI 오류 코드와 분리한다.

## 4) Integration & Verification (통합 및 검증)

### 4.1 통합 경로 정의

| 경로 단계 | 컴포넌트 | 책임 | 의존성 방향 | 검증 방법 |
|---|---|---|---|---|
| I1 | Screen Boundary | 입력 계약 검증/응답 포맷 | Screen -> Application | Boundary 통합 테스트 |
| I2 | Application Service(선택) | 유스케이스 오케스트레이션 | Application -> Domain, Data | 오케스트레이션 테스트 |
| I3 | Domain | 규칙 판정/결과 결정 | Domain (독립) | 도메인 통합 테스트 |
| I4 | Data Repository | 저장/로드 | Application -> Data Interface | 저장 연계 테스트 |

의존성 원칙:
- 내부 계층(Domain)은 외부 계층(UI/Data)을 참조하지 않는다.
- Screen/Data는 Domain 계약에 의존한다.
- 구현체 결합은 Composition Root에서만 허용한다.

체크리스트:
- [ ] 의존성 역전 원칙을 구조적으로 보장한다.
- [ ] Domain 단독 테스트 가능 상태를 유지한다.
- [ ] Application 제거 시에도 Screen-Domain 직접 연결이 가능해야 한다.

## 즉시 RED로 시작할 첫 10개 테스트

1. 4x4가 아닌 입력(예: 3x4)은 UI-ERR-001을 반환해야 한다.
2. 값에 17이 포함되면 UI-ERR-002를 반환해야 한다.
3. 빈칸(0)이 1개이면 UI-ERR-003을 반환해야 한다.
4. 0 제외 중복 값이 있으면 UI-ERR-004를 반환해야 한다.
5. 유효 입력에서 빈칸 좌표를 정확히 2개(1-index 기준) 식별해야 한다.
6. 유효 입력에서 누락 숫자 2개를 오름차순으로 도출해야 한다.
7. 작은수->첫 빈칸 조합이 성립하면 해당 순서로 int[6]을 반환해야 한다.
8. 작은수->첫 빈칸 조합이 실패하면 반대 조합으로 int[6]을 반환해야 한다.
9. 성공 결과는 반드시 길이 6, 좌표는 1~4 범위를 만족해야 한다.
10. 저장소(File) 저장 후 로드한 입력/결과가 원본과 완전히 동일해야 한다.
